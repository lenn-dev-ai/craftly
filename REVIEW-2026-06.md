## Executive Summary

## A) Security Findings

### CRITICAL

#### A1.1 — `app/api/cron/keep-alive/route.ts:13-22` — Offener Endpoint bei fehlendem CRON_SECRET
**Auth-Status:** CRITICAL  
**Beschreibung:** Die `keep-alive`-Route prüft `CRON_SECRET` nur wenn die Env-Var gesetzt ist (`if (cronSecret) { ... }`). Ist `CRON_SECRET` in der Netlify-Config nicht gesetzt, ist der Endpoint vollständig offen für anonyme GET-Requests. Der Handler instantiiert dann `createServiceRoleClient()` und liest die `profiles`-Tabelle. Da `CRON_SECRET` kein Pflichtfeld ist, kann diese Situation in jedem Deployment auftreten.

```ts
// Problematischer Code:
if (cronSecret) {
  const authViaSecret = request.headers.get("x-cron-secret") === cronSecret
  if (!authViaSecret) { return 401 }
}
// Wenn cronSecret undefined: dieser Block wird übersprungen → offen
const supabase = createServiceRoleClient()  // ← Service-Role ohne Auth!
```

**Empfehlung:** Invertieren: wenn `CRON_SECRET` NICHT gesetzt ist → sofort 503 ("Endpoint not configured"). Nur wenn Secret vorhanden und Wert stimmt → weiter.

### HIGH

#### A1.2 — `app/api/melden/pills/route.ts:124` — GET-Endpoint ohne Auth, nutzt Service-Role-Client
**Auth-Status:** HIGH  
**Beschreibung:** Die Route ist bewusst "offen" (laut Kommentar: Pills sind keine sensitiven Daten), aber sie nutzt `createServiceRoleClient()` für die Verwalter-Statistik-Query und lädt bis zu 500 Ticket-Zeilen pro Request. Bei `wohnung_id`-Parameter werden zwei DB-Queries ausgeführt (wohnungen + tickets). Ohne Auth ist das ein potenzieller DoS-Vektor via massenhafte Requests mit verschiedenen wohnung_ids.  
**Empfehlung:** Entweder Auth hinzufügen (soft: jeder logged-in User), oder auf `createServerSupabaseClient()` wechseln (der via RLS sicherer ist). Aktuellen `createServiceRoleClient()`-Aufruf durch einen authentifizierten Client ersetzen.

#### A2.1 — RLS `profiles`: Sensible Felder für alle authentifizierten User lesbar
**Migration:** `supabase/migrations/20260519000000_security_hardening.sql:108-110`  
**Beschreibung:** `profiles_select` Policy lautet `USING (auth.uid() IS NOT NULL)` — d.h. jeder eingeloggte User (auch ein Mieter mit Zugang zur App) kann `SELECT *` auf beliebige Profile machen und sieht dann: `telefon`, `stripe_account_id`, `stripe_charges_enabled`, `early_adopter_bis`, `angebotstreue`, `bewertung_avg`, `plz_bereich`. Das ist wahrscheinlich nicht intentional. Früher war es `USING (true)` (anonym), jetzt ist es auth-gated — aber die Granularität fehlt.  
**Empfehlung:** Column-level Security ist in Postgres-RLS nicht nativ möglich, daher: Either (a) eigenes Profil vollständig, fremde Profile nur projizierte Subset-View (name, firma, gewerk, bewertung_avg — keine telefon/stripe/early_adopter-Felder), oder (b) DB-View `profiles_public` mit nur öffentlichen Feldern.

#### A2.2 — `diagnose_preise`: `USING (true)` für alle authentifizierten User
**Migration:** `supabase/migrations/20260101000000_diagnose_projekt.sql:60-62`  
**Beschreibung:** `diagnose_preise_select_authenticated` Policy nutzt `USING (true)` für alle `authenticated` User. Mieter und Handwerker können damit interne Preisparameter der Diagnose-Gebühren lesen (z.B. Stundensatz-Basis pro Gewerk). Kein direkter Exploit, aber Information Disclosure.  
**Empfehlung:** Policy auf `USING (public.is_verwalter() OR public.is_admin())` einschränken — nur Verwalter brauchen diese Preise beim Diagnose-Ticketing.

#### A3.1 — Doppelte API-Routes `api/diagnose/*` als Wrapper
**Dateien:** `app/api/diagnose/befund-abgeben/route.ts`, `projekt-annehmen/route.ts`, `projekt-zur-auktion/route.ts`, `termin-annehmen/route.ts`

#### A1.4 — X-Frame-Options Konflikt
**Dateien:** `next.config.js:10`, `netlify.toml:19`  
**Beschreibung:** `next.config.js` setzt `X-Frame-Options: DENY`, `netlify.toml` setzt `X-Frame-Options: SAMEORIGIN`. Netlify-Edge-Layer überschreibt den Next.js-Header. Effektiv gilt `SAMEORIGIN` in Production — die strengere `DENY`-Konfiguration bleibt wirkungslos.  
**Empfehlung:** `next.config.js` Header entfernen oder auf `SAMEORIGIN` vereinheitlichen. `netlify.toml` ist die Quelle der Wahrheit für Edge-Headers.

#### A1.5 — CSP unvollständig in `netlify.toml:21`
**Beschreibung:** `connect-src` enthält nur `self`, Supabase-Domains und `photon.komoot.io`. Folgende aktiv genutzte externe Domains fehlen:

- `https://*.mapbox.com` / `https://events.mapbox.com` — Mapbox Tiles + Telemetry
- `https://*.sentry.io` / `https://browser.sentry-cdn.com` — Sentry Error-Reporting
- `https://api.vapi.ai` — Vapi Voice-AI (wenn im Frontend genutzt)

Client-side Stripe-Operations können scheitern; Sentry-Fehler werden in Production nicht übertragen.  
**Empfehlung:** CSP `connect-src` um die oben genannten Domains erweitern. `img-src` um `https://api.mapbox.com` (Mapbox Static Tiles) ergänzen.

## B) Performance Findings

### B1 — Build-Output (Next.js 14 Build)

**Build-Status:** Kompilierung erfolgreich (`✓ Compiled successfully`). Webpack-Warning: `Serializing big strings (231kiB) impacts deserialization performance` — ein Client-Bundle serialisiert einen sehr großen String (wahrscheinlich der JSON-LD-Block in `app/layout.tsx` oder ein inline-SVG).

**Bundle-Größen (aus Build-Analyse):**
- Haupt-Page (`app/page.tsx`): Server-seitig gerendert, 40kB Server-JS — akzeptabel.

**Schätzung First-Load-JS für `/dashboard-verwalter/marktplatz`:** > 400kB (Mapbox + React + App-Code). Das ist über dem 150kB-Flag.

**Empfehlung:** Mapbox in `marktplatz/page.tsx` via `dynamic()` mit `ssr: false` lazy-loaden, so wie es `karte/page.tsx` bereits macht.

### B2 — Client-Components Analyse

**Anzahl `"use client"` Files:** 66 (aus `app/` + `components/`)

**Identifizierte Server-Component-Kandidaten:**

| File | Problem | Empfehlung |
|------|---------|------------|
| `app/dashboard-handwerker/auftraege/page.tsx` | `useEffect` für Daten-Fetch, kein State nötig außer für Filter | Server-Component mit `supabase.from("tickets").select(...)` + Client-Component nur für interaktive Filter |

### B3 — N+1 / Batching Probleme

#### B3.1 — `app/api/cron/sichtbarkeits-recompute/route.ts:63-122` — Kritisches N+1-Pattern
**Beschreibung:** Der Cron macht für jeden Handwerker **4 separate DB-Round-Trips** in einem `for`-Loop:
1. `hw_google_oauth` SELECT (1 Query, nicht batched)
2. `einladungen` (×2) + `stamm_anfragen` (×2) — 4 Queries via `Promise.all` 
3. `einladungen` + `stamm_anfragen` angenommen — 2 Queries via `Promise.all`
4. `profiles` UPDATE

Das sind **ca. 4 Wait-Gruppen × N HW**. Selbst bei 50 HW: ~200 DB-Round-Trips sequenziell. Bei 200 HW: ~800 Round-Trips, Laufzeit 30-120s (Netlify Function-Timeout: 10-26s auf Free-Tier, 26s auf Pro).

**Kommentar im Code:** "Bei 100-1000 HW noch eine Update pro HW akzeptabel (≤ 2s Latenz). Bei mehr: SQL-Function wäre nächster Schritt."

**Aktueller Scale:** Da es sich um eine Beta handelt, ist die Anzahl der HW noch niedrig. Aber das Risiko ist bekannt und dokumentiert.

**Empfehlung (nächster Sprint):** Alle 4 Queries in einer PostgreSQL-Function zusammenfassen (`recompute_sichtbarkeit_all()`), die per Batch über alle HW iteriert. Das reduziert von N×4 Round-Trips auf 1 RPC-Call.

#### B3.2 — `app/dashboard-verwalter/marktplatz/page.tsx:155-180` — Batching vorhanden, aber suboptimal
**Beschreibung:** Die Seite lädt zunächst `tickets`, dann separate Profile für die HW-IDs via `.in("id", uniqueIds)`. Das ist korrektes Batching. Jedoch: `stamm_handwerker` und Pool-Queries sind separate Calls, die nicht zusammen batched werden.  
**Empfehlung:** Nicht kritisch, niedrige Priorität.

#### B3.3 — `app/api/verwalter/kpis/route.ts` — Client-seitiges JS-Aggregat statt SQL
**Beschreibung:** Die KPI-Route lädt alle Tickets der letzten 28 Tage und berechnet Aggregationen (Wochen-Throughput, Erledigt-Rate) in JavaScript. Kommentar: "PostgREST hat keine elegante GROUP-BY-Woche-Syntax." Das ist ein Bandwidth-Problem: alle Ticket-Rows werden über das Netzwerk geschickt, wenn eine SQL-Aggregation (RPC) viel effizienter wäre.  
**Empfehlung:** Mittlere Priorität — bei >1000 Tickets/Verwalter wird das zum Problem.

## C) Architektur/Konsistenz Findings

### C1 — TypeScript: `as any` Verwendungen

**Stellen:**
- `app/dashboard-handwerker/angebot/[id]/page.tsx:383`: `(ticket.objekte as any).name` — `objekte` ist ein Join-Result ohne typisierten Rückgabetyp
- `components/ticket/TicketDetailView.tsx:22`: `(angebot.handwerker as any)?.bewertung_avg`
- `components/ticket/TicketDetailView.tsx:966,968,969,1105`: 5× `as any` auf `hw.handwerker` und `a.handwerker` Join-Ergebnisse
- `components/ticket/TicketDetailView.tsx:1153`: `(a as any).geschaetzte_dauer`

**Pattern:** Join-Ergebnisse von Supabase (`.select("..., handwerker:profiles(...)")`) sind ohne explizite TypeScript-Interface nicht typisiert. Das führt zu `as any`-Casts.

**Empfehlung:** Join-Typen als Interface definieren:
```ts
interface AngebotMitHandwerker {
  id: string; preis: number; /* ... */
  handwerker: { name: string | null; firma: string | null; bewertung_avg: number | null }
}
```

### C2 — Inkonsistentes Error-Handling in API-Routes

Alle API-Routes geben `{ error: "..." }` mit HTTP-Status zurück — konsistent. Eine Ausnahme:  
- `app/api/cron/keep-alive/route.ts:24`: `if (error) throw error` — der einzige `throw` in einer API-Route. Bei einem DB-Error wird eine unbehandelte Exception geworfen, die Next.js als 500 mit Stack-Trace behandelt (kein strukturiertes JSON).

**Empfehlung:** `throw error` durch `return NextResponse.json({ ok: false, error: String(error) }, { status: 500 })` ersetzen.

### C3 — Middleware Auth-Pattern: Cookie-Pre-Filter statt `getUser()`

**Datei:** `middleware.ts:75-84`  
**Beschreibung:** Die Middleware nutzt einen "Cookie-Pre-Filter" statt `supabase.auth.getUser()` für den primären Auth-Check:
```ts
const hasAuthCookie = request.cookies.getAll()
  .some(c => c.name.startsWith("sb-") && c.name.includes("auth-token"))
```
Das bedeutet: abgelaufene oder ungültige JWT-Tokens werden von der Middleware durchgelassen. Der echte Auth-Check passiert erst im `RoleGuard` auf der Client-Seite. Die Motivation (Login-Loop-Vermeidung nach `signInWithPassword`) ist kommentiert und valide für den Browser-Flow. API-Routes sind von diesem Verhalten nicht betroffen (sie laufen ihren eigenen `getUserFromRequest`-Check).

**Empfehlung:** Kein Fix nötig für den aktuellen MVP-Stand, aber dokumentieren dass API-Routes die einzige vertrauenswürdige Auth-Schicht sind.

### C4 — `lib/yield-management.ts` — Vollständig toter Code

## Methodische Anmerkungen

- **Build Output:** `npm run build` abgebrochen nach `Compiled successfully` während TypeScript-Check (Laufzeit >15 Min auf lokalem System — wahrscheinlich Cache-Invalidierung nach mehreren konkurrierenden Build-Prozessen). Webpack-Warning "Serializing big strings (231kiB)" wurde erfasst. Bundle-Größen konnten nicht exakt gemessen werden.
- **Lighthouse:** `npx lighthouse --version: 13.4.0` vorhanden, aber kein Chromium-Browser auf macOS gefunden. Performance-Schätzungen basieren auf Code-Analyse + bekannten Bundle-Größen der Abhängigkeiten (Mapbox ~250-400kB, Sentry ~20-40kB, Recharts ~80kB).
- **Supabase RLS Remote:** RLS-Analyse basiert ausschließlich auf den Migration-Files. Ob die neuesten Migrations in Production deployed sind, wurde nicht via MCP-Supabase geprüft.
- **`stamm_handwerker` RLS:** Tabelle hat kein Migration-File im Repo. RLS-Status in Production unbekannt — muss via Supabase-Dashboard oder `list_tables` MCP geprüft werden.
